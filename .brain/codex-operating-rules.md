# Codex Operating Rules

1. Read the `.brain` files before making meaningful changes.
2. Inspect existing code paths before proposing new patterns.
3. Prefer extending existing systems over introducing new abstractions.
4. Do not treat this product as a generic enterprise PM suite.
5. Preserve simplicity for student users and classroom workflows.
6. Prioritize cognitive scaffolding over feature density.
7. Do not make schema or architecture changes silently; call them out explicitly.
8. Avoid unnecessary dependencies and framework churn.

## Execution protocol
For non-trivial tasks:
1. Summarize your understanding of the request and context.
2. List files likely to be affected.
3. Identify key risks and tradeoffs.
4. Propose a concrete implementation plan.
5. Implement in small, verifiable steps.

After implementation:
- Summarize what changed.
- List touched files.
- Note tradeoffs, limitations, and follow-up items.
